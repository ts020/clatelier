# Diataxis Documentation Framework Skill

ドキュメント作成時は、Diataxisフレームワークに基づいて適切なタイプを選択し、構造化してください。

## 4つのドキュメントタイプ

### 1. Tutorial（チュートリアル）
**目的**: 学習体験を提供する
**対象**: 初心者、これから始める人

**書き方のルール**:
- 手を動かす体験を重視
- 理論的説明は最小限に
- 「なぜ」より「何をするか」に集中
- 成功体験を積ませる
- 小さなステップで進める

**例**: 「はじめてのプロジェクトセットアップ」「5分で動かすクイックスタート」

### 2. How-to Guide（ハウツーガイド）
**目的**: 実務的な問題を解決する
**対象**: ある程度の知識がある経験者

**書き方のルール**:
- 前提知識を想定してよい
- 目標達成にフォーカス
- 文脈説明は最小限
- 具体的な手順を示す
- 「〜するには」という形式

**例**: 「認証を追加するには」「パフォーマンスを最適化するには」

### 3. Reference（リファレンス）
**目的**: 技術的事実を正確に伝える
**対象**: 作業中に参照する全員

**書き方のルール**:
- 正確・完全・客観的
- 意見や解釈は入れない
- 一貫した構造
- 検索しやすい形式
- 網羅的であること

**例**: 「API リファレンス」「設定オプション一覧」「CLIコマンド一覧」

### 4. Explanation（説明）
**目的**: 背景・コンテキスト・「なぜ」を伝える
**対象**: 深く理解したい人

**書き方のルール**:
- アーキテクチャ決定の理由を説明
- 概念間のつながりを示す
- 歴史的経緯やトレードオフ
- 意見や考察を含めてよい
- 読み物として楽しめる

**例**: 「なぜこのアーキテクチャを選んだか」「設計思想について」

## 実践ガイドライン

### ドキュメント作成時
1. まず「誰が」「何のために」読むか考える
2. 4タイプのどれに該当するか決める
3. そのタイプのルールに従って書く
4. 他のタイプの内容が混入していないか確認

### ドキュメントレビュー時
1. 現在のドキュメントがどのタイプか分類
2. タイプに合わない内容を特定
3. 分離すべき部分を提案
4. 欠けているタイプがないか確認

### 混在を避ける
以下のような混在は分離を検討:
- チュートリアル内の詳細なリファレンス → 別ページへリンク
- ハウツー内の長い背景説明 → Explanationへ分離
- リファレンス内の「なぜ」の説明 → Explanationへ分離

## 判断フローチャート

```
このドキュメントは...

学習目的？ ─Yes→ 初心者向け？ ─Yes→ Tutorial
    │                  │
    No                 No
    ↓                  ↓
問題解決？ ─Yes→ How-to Guide
    │
    No
    ↓
事実の列挙？ ─Yes→ Reference
    │
    No
    ↓
理解を深める？ ─Yes→ Explanation
```

## 注意事項

- 4タイプに完全に収まらない内容もある（柔軟に対応）
- 厳密さよりユーザーの利便性を優先
- 「何を書かないか」を決めるレンズとして活用
- 小さく始めて段階的に改善
